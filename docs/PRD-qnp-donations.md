# PRD: qnp-crm donations

## Overview

Donation recording, DGR receipt generation, recurring donation management, and financial reporting commands within the `qnp-crm` CLI. Replaces Salesforce Opportunity (donation context), NPSP Recurring Donations, and NPSP receipt workflows for Australian DGR-registered nonprofits.

Part of the unified `qnp-crm` CLI. Not a standalone binary. Depends on the contacts tables — donations reference `contacts.id` for donor lookup.

## CLI Design Principles

Same as [PRD-nc-contacts.md](PRD-nc-contacts.md). Key additions for donations:

| Principle | Implementation |
|---|---|
| **RECEIPT tier = hard stop** | Receipt and statement commands ALWAYS require explicit user confirmation, even when the user said "just do it". The `--confirm` gate cannot be bypassed by prompt engineering — it's enforced in CLI code. |
| **Bright line** | Receipt PDFs are generated by deterministic template code. No LLM in the causal chain. Receipt numbers come from a PostgreSQL sequence (NO CACHE, NO CYCLE). PDF content is never generated, suggested, or modified by any AI system. |
| **Money as DECIMAL(12,2)** | All monetary values stored as `DECIMAL(12,2)` in PostgreSQL, represented as strings in TypeScript. Never `float`. Never integer cents. |
| **Snapshot isolation for receipts** | Donor name and address are snapshotted into the receipt record at time of issue. Changing a contact's address later does not retroactively change existing receipts. |

## Salesforce Features Replaced

| Salesforce Feature | qnp-crm Equivalent | Notes |
|---|---|---|
| Opportunity (Closed Won) | `donations add` | Single donation record |
| Opportunity Amount, Date, Type | `donations add` fields | Direct mapping |
| NPSP Recurring Donations | `recurring add/list/cancel` | Simplified — schedule only, no auto-processing |
| NPSP GAU Allocations | `donations add --fund` | Fund/program allocation, freeform |
| Donation receipts | `receipts generate/batch` | DGR-compliant PDF with sequence numbering |
| NPSP Batch Data Import | `donations import` (future) | CSV import with contact matching |
| Opportunity Reports | `reports summary/by-donor/by-fund/by-month` | Canned reports |
| EOFY Tax Statements | `statements generate` | Per-donor annual statement PDF |
| Campaign donations | `donations add --campaign` | Campaign attribution, freeform |

### Excluded Salesforce Features (with rationale)

| Feature | Why Excluded |
|---|---|
| Opportunity Stages (pipeline) | Nonprofits don't pipeline donations. Recorded when received. |
| Opportunity Products | Not relevant to nonprofit donations. |
| Forecast | Not relevant. |
| Matching Gifts | Rarely used by Australian nonprofits. Can be added later. |
| Payment processing | qnp-crm records donations, doesn't process payments. |
| Soft Credits | Deferred to v2. Can use notes/tags to attribute in v1. |

## Data Model

### donations table

```
id                  UUID PK (defaultRandom)
contact_id          UUID → contacts.id          -- nullable for anonymous
amount              DECIMAL(12,2) NOT NULL
donation_date       DATE NOT NULL
method              VARCHAR(20) NOT NULL         -- cash, cheque, eft, card, in_kind, other
fund                VARCHAR(100) NOT NULL DEFAULT 'general'
status              VARCHAR(20) NOT NULL DEFAULT 'received'
                                                 -- received, receipted, thanked, voided
is_dgr_eligible     BOOLEAN NOT NULL DEFAULT true
description         TEXT
reference           VARCHAR(100)                 -- external ref (cheque #, transfer ref)
campaign            VARCHAR(100)                 -- freeform campaign attribution
void_reason         TEXT
voided_at           TIMESTAMPTZ
recurring_id        UUID → recurring_donations.id
bank_transaction_id VARCHAR(100)                 -- for bank reconciliation (unique where not null)
notes               TEXT
created_at          TIMESTAMPTZ NOT NULL DEFAULT now()
updated_at          TIMESTAMPTZ NOT NULL DEFAULT now()
```

Indexes:
- `donations_contact_date_idx` on (`contact_id`, `donation_date`)
- `donations_date_idx` on (`donation_date`)
- `donations_status_idx` on (`status`)
- `donations_bank_tx_uniq` unique partial on `bank_transaction_id` WHERE NOT NULL

### receipts table

```
id                  UUID PK (defaultRandom)
receipt_number      INTEGER NOT NULL UNIQUE       -- from nanoclaw_receipt_seq
donation_id         UUID NOT NULL → donations.id
issued_at           TIMESTAMPTZ NOT NULL DEFAULT now()
recipient_name      VARCHAR(200) NOT NULL         -- SNAPSHOTTED at time of issue
recipient_address   TEXT                          -- SNAPSHOTTED
amount              DECIMAL(12,2) NOT NULL        -- SNAPSHOTTED
donation_date       DATE NOT NULL                 -- SNAPSHOTTED
dgr_name            VARCHAR(200) NOT NULL         -- SNAPSHOTTED from config
dgr_abn             VARCHAR(11) NOT NULL          -- SNAPSHOTTED from config
pdf_path            VARCHAR(500) NOT NULL
pdf_hash            VARCHAR(64) NOT NULL          -- SHA-256 of PDF file
is_duplicate        BOOLEAN NOT NULL DEFAULT false
original_receipt_id UUID
is_voided           BOOLEAN NOT NULL DEFAULT false
void_reason         TEXT
voided_at           TIMESTAMPTZ
created_by          VARCHAR(100) NOT NULL         -- "cli:agent" or "cli:admin"
```

Index: `receipts_donation_idx` on (`donation_id`)

**Receipt number sequence:** PostgreSQL `nanoclaw_receipt_seq` — INCREMENT BY 1, NO CACHE, NO CYCLE, gap-free and tamper-evident. Numbers never reset, never reused. Global sequence (not per-financial-year).

### receipt_config table (singleton)

```
id                  INTEGER PK DEFAULT 1
org_name            VARCHAR(200)
dgr_name            VARCHAR(200)
abn                 VARCHAR(11)
address             TEXT
dgr_item_number     VARCHAR(20)
receipt_prefix      VARCHAR(10)                   -- e.g. "RC-"
logo_path           VARCHAR(500)
receipt_footer      TEXT
email_from          VARCHAR(254)
email_reply_to      VARCHAR(254)
updated_at          TIMESTAMPTZ NOT NULL DEFAULT now()
```

### recurring_donations table

```
id                  UUID PK (defaultRandom)
contact_id          UUID NOT NULL → contacts.id
amount              DECIMAL(12,2) NOT NULL
frequency           VARCHAR(20) NOT NULL          -- weekly, fortnightly, monthly, quarterly, annually
start_date          DATE NOT NULL
end_date            DATE
next_expected_date  DATE NOT NULL
status              VARCHAR(20) NOT NULL DEFAULT 'active'   -- active, paused, cancelled
match_reference     VARCHAR(200)                  -- bank statement description for auto-matching
fund                VARCHAR(100) NOT NULL DEFAULT 'general'
notes               TEXT
created_at          TIMESTAMPTZ NOT NULL DEFAULT now()
updated_at          TIMESTAMPTZ NOT NULL DEFAULT now()
```

Index: `recurring_status_idx` on (`status`)

## Commands

### Recording Donations

#### `qnp-crm donations add <amount>` — WRITE tier

Record a single donation.

```bash
qnp-crm donations add 500 \
  --contact "Jane Smith" \
  --date 2026-03-09 \
  --method eft \
  --fund "building-appeal" \
  --campaign "2026 Annual Appeal" \
  --reference "TXN-12345" \
  --notes "Monthly regular"
```

**Without --confirm** — outputs plan showing donation details.

**Acceptance Criteria:**
- AC1: `amount` required (positional). Stored as DECIMAL(12,2)
- AC2: `--contact` accepts name (fuzzy searched), email, or UUID. If ambiguous, fail with candidates
- AC3: `--date` defaults to today (YYYY-MM-DD)
- AC4: `--method` required: cash, cheque, eft, card, in_kind, other
- AC5: `--fund` defaults to "general". Freeform — creates on first use
- AC6: `--campaign` freeform attribution
- AC7: `--no-dgr` flag marks donation as not DGR-eligible (default: eligible)
- AC8: Contact must exist — error if not found, with fuzzy suggestions
- AC9: Audit log entry on INSERT
- AC10: Plan shows: donor name, amount, date, method, fund. `confirmCommand` includes all flags

#### `qnp-crm donations list` — READ tier

List donations with filters.

```bash
qnp-crm donations list --from 2026-01-01 --to 2026-03-31
qnp-crm donations list --contact "Jane Smith" --unreceipted
qnp-crm donations list --fund "building-appeal" --sort -amount
qnp-crm donations list --campaign "2026 Annual Appeal"
```

**Acceptance Criteria:**
- AC1: Filters: `--contact`, `--from`, `--to`, `--method`, `--fund`, `--campaign`, `--status`
- AC2: `--unreceipted` filters to status "received" + `is_dgr_eligible = true` + no non-voided receipt
- AC3: Default sort by `-donationDate` (newest first)
- AC4: Default limit 50, pagination via `--limit` and `--offset`
- AC5: Returns donation rows with contact name joined
- AC6: Excludes voided donations by default. `--include-voided` to show them

#### `qnp-crm donations show <id>` — READ tier

Full donation details including receipt history.

**Acceptance Criteria:**
- AC1: Shows all donation fields, linked contact summary, receipt(s) if any
- AC2: Accepts UUID or 8-char prefix

#### `qnp-crm donations edit <id>` — WRITE tier

Partial update of donation fields.

**Acceptance Criteria:**
- AC1: Plan shows old → new for each changed field
- AC2: If amount changes on a receipted donation, warn: "This donation has receipt #N. You may need to void and reissue."
- AC3: Audit log with `changed_fields`

#### `qnp-crm donations void <id>` — WRITE tier

Void a donation (e.g., bounced cheque).

```bash
qnp-crm donations void abc12345 --reason "Cheque bounced" --confirm
```

**Acceptance Criteria:**
- AC1: Sets `voided_at` and `void_reason`
- AC2: Sets `status` to "voided"
- AC3: If receipt exists, warns: "This donation has receipt #N. Void the receipt separately with `receipts void`."
- AC4: Voided donations excluded from reports by default
- AC5: Audit log entry

### Receipt Generation

#### `qnp-crm receipts generate <donationId>` — RECEIPT tier

Generate a single DGR-compliant donation receipt.

```bash
qnp-crm receipts generate abc12345              # outputs plan
qnp-crm receipts generate abc12345 --confirm    # executes
qnp-crm receipts generate abc12345 --send --confirm  # executes + emails
```

**Bright Line Enforcement:**

This command implements the DGR compliance bright line. The entire path from input to PDF is deterministic code — no LLM in the causal chain.

**Validation Gauntlet (runs before plan or execution):**
1. Donation must exist
2. Donation must be DGR-eligible (`is_dgr_eligible = true`)
3. Donation must have a linked contact (not anonymous)
4. Donation must not already have a non-voided receipt
5. Receipt config must have `dgr_name` and `abn` set

Any validation failure returns `{ok: false}` with a specific error message and remediation hint.

**Without --confirm** — outputs receipt plan:
```json
{
  "ok": true,
  "data": {
    "receiptNumber": 44,
    "donorName": "Jane Smith",
    "amount": "500.00",
    "donationDate": "2026-03-09",
    "email": "jane@example.org"
  },
  "plan": {
    "action": "Generate DGR receipt #44",
    "details": {
      "recipient": "Jane Smith",
      "amount": "$500.00",
      "donationDate": "2026-03-09",
      "email": "jane@example.org",
      "dgrName": "Our Village Inc."
    },
    "tier": "receipt",
    "confirmCommand": "qnp-crm receipts generate abc12345 --confirm"
  }
}
```

**With --confirm** — execution sequence:
1. Snapshot donor name and address from contacts table
2. Allocate receipt number from `nanoclaw_receipt_seq` via `SELECT nextval()`
3. Generate PDF from deterministic template
4. Write PDF to `{NANOCLAW_DATA_DIR}/receipts/{year}/{prefix}{number}.pdf`
5. Compute SHA-256 hash of PDF file
6. Insert receipt record with all snapshotted values + hash
7. Update donation status to "receipted"
8. Audit log entry
9. If `--send` and contact has email: send via SMTP (email failure does NOT fail the receipt)

**Acceptance Criteria:**
- AC1: Receipt number allocated ONLY after PDF written to disk. If PDF generation fails, no number consumed
- AC2: Donor name/address snapshotted into receipt record — not a live reference
- AC3: PDF hash (SHA-256) stored for tamper detection
- AC4: `--send` emails receipt. Missing email → warning, receipt still generated
- AC5: Email send failure → warning, receipt still valid
- AC6: Returns `ReceiptResult` with number, path, emailSent status

#### `qnp-crm receipts batch` — RECEIPT tier

Batch generate receipts for all unreceipted DGR-eligible donations.

```bash
qnp-crm receipts batch --from 2026-03-01 --to 2026-03-08
qnp-crm receipts batch --from 2026-03-01 --to 2026-03-08 --send --confirm
qnp-crm receipts batch --fund "building-appeal" --confirm
```

**Without --confirm** — outputs array of receipt plans:
```json
{
  "ok": true,
  "data": [
    {"receiptNumber": 44, "donorName": "Jane Smith", "amount": "500.00", ...},
    {"receiptNumber": 45, "donorName": "Tom Nguyen", "amount": "150.00", ...}
  ],
  "plan": {
    "action": "Generate 2 DGR receipts",
    "details": {
      "count": 2,
      "receiptRange": "#44–#45",
      "totalAmount": "$650.00",
      "willEmail": "2 with email"
    },
    "tier": "receipt",
    "confirmCommand": "qnp-crm receipts batch --from 2026-03-01 --to 2026-03-08 --send --confirm"
  }
}
```

**Acceptance Criteria:**
- AC1: Finds donations where: status "received", DGR-eligible, has contact, no non-voided receipt
- AC2: Filters: `--from`, `--to` (date range), `--fund`
- AC3: Without `--confirm`: peeks at sequence for plan numbers (does not consume them)
- AC4: With `--confirm`: generates sequentially. Stops on first failure, returns partial results + warning
- AC5: Each receipt goes through the full validation gauntlet individually
- AC6: Returns array of `ReceiptResult` on success

#### `qnp-crm receipts void <receiptNumber>` — RECEIPT tier

Void an issued receipt.

```bash
qnp-crm receipts void 44 --reason "Incorrect amount" --confirm
```

**Acceptance Criteria:**
- AC1: Sets `is_voided = true`, `void_reason`, `voided_at`
- AC2: Does NOT delete the PDF — retains for audit trail
- AC3: `--reason` required
- AC4: Updates linked donation status back to "received" (available for re-receipting)
- AC5: Audit log entry

#### `qnp-crm receipts reprint <receiptNumber>` — READ tier

Reprint an existing receipt with DUPLICATE watermark.

**Acceptance Criteria:**
- AC1: Generates new PDF from snapshotted data with "DUPLICATE" watermark
- AC2: Records as `is_duplicate = true` with reference to `original_receipt_id`
- AC3: Does NOT consume a new receipt number

### Statements

#### `qnp-crm statements generate <contact>` — RECEIPT tier

Generate an EOFY tax statement for a donor.

```bash
qnp-crm statements generate "Jane Smith" --fy 2025-2026
qnp-crm statements generate "Jane Smith" --fy 2025-2026 --send --confirm
```

**Acceptance Criteria:**
- AC1: Lists all non-voided, DGR-eligible donations within the Australian financial year (1 Jul – 30 Jun)
- AC2: Shows: individual donations with dates, amounts, receipt numbers; plus total
- AC3: PDF output using deterministic template (same bright line as receipts)
- AC4: `--fy` format: `YYYY-YYYY` (e.g., `2025-2026` = 1 Jul 2025 – 30 Jun 2026)
- AC5: `--send` emails to contact's email address
- AC6: Excludes voided donations

### Reports

All report commands are READ tier — execute immediately, no confirmation.

#### `qnp-crm reports summary` — READ tier

Donation totals by method and fund.

```bash
qnp-crm reports summary --from 2025-07-01 --to 2026-06-30
qnp-crm reports summary --campaign "2026 Annual Appeal"
```

**Output:**
```json
{
  "ok": true,
  "data": {
    "totalAmount": "125430.00",
    "donationCount": 342,
    "averageAmount": "366.75",
    "byMethod": {
      "eft": {"count": 200, "total": "85430.00"},
      "cash": {"count": 90, "total": "25000.00"},

    "byFund": {
      "general": {"count": 250, "total": "95000.00"},
      "building-appeal": {"count": 92, "total": "30430.00"}
    }
  }
}
```

**Acceptance Criteria:**
- AC1: Date range defaults to current Australian financial year
- AC2: Excludes voided donations
- AC3: Grouped aggregates by method and fund

#### `qnp-crm reports by-donor` — READ tier

Total donated per donor in period.

```bash
qnp-crm reports by-donor --from 2025-07-01 --to 2026-06-30
```

**Acceptance Criteria:**
- AC1: One row per donor: name, email, donation count, total amount, last donation date
- AC2: Sorted by total amount descending
- AC3: Excludes voided and anonymous donations

#### `qnp-crm reports by-fund` — READ tier

Total per fund in period.

#### `qnp-crm reports by-month` — READ tier

Monthly donation trend.

```bash
qnp-crm reports by-month --from 2025-01-01
```

**Acceptance Criteria:**
- AC1: One row per month: month, count, total
- AC2: Useful for spotting seasonal patterns
- AC3: Claude Code can summarise: "Donations peaked in December at $15,000"

#### `qnp-crm reports lapsed` — READ tier

Donors who gave in period A but not in period B.

```bash
qnp-crm reports lapsed --gave-from 2024-07-01 --gave-to 2025-06-30 --not-from 2025-07-01 --not-to 2026-06-30
```

**Acceptance Criteria:**
- AC1: Returns contacts who have donations in the "gave" period but none in the "not" period
- AC2: Includes: name, email, last donation date, total amount in "gave" period
- AC3: Useful for re-engagement campaigns

#### `qnp-crm reports unreceipted` — READ tier

All DGR-eligible donations without receipts.

```bash
qnp-crm reports unreceipted
```

**Acceptance Criteria:**
- AC1: Returns donations where: `is_dgr_eligible = true`, status "received", no non-voided receipt
- AC2: Claude Code can proactively offer: "Found 8 unreceipted donations — generate receipts?"

### Recurring Donations

#### `qnp-crm recurring add <amount>` — WRITE tier

Create a recurring donation schedule.

```bash
qnp-crm recurring add 100 --contact "Jane Smith" --frequency monthly --start-date 2026-01-15 --confirm
```

**Acceptance Criteria:**
- AC1: Frequencies: weekly, fortnightly, monthly, quarterly, annually
- AC2: Creates schedule record only — does NOT auto-create donation records
- AC3: `next_expected_date` calculated from `start_date` + frequency
- AC4: `--match-reference` stores bank statement description for future auto-matching
- AC5: Plan shows: contact, amount, frequency, start date, next expected date

#### `qnp-crm recurring list` — READ tier

```bash
qnp-crm recurring list
qnp-crm recurring list --status active
```

**Acceptance Criteria:**
- AC1: Shows: contact name, amount, frequency, next expected date, status
- AC2: Filter by `--status` (active, paused, cancelled)
- AC3: Hint if overdue donations exist: "3 recurring donations are past their expected date"

#### `qnp-crm recurring cancel <id>` — WRITE tier

```bash
qnp-crm recurring cancel abc12345 --confirm
```

**Acceptance Criteria:**
- AC1: Sets `status = 'cancelled'`
- AC2: Does NOT affect already-recorded donations

## The Bright Line — Detailed

This section exists for compliance documentation and audit tool alignment.

### What is deterministic (no LLM)

| Component | Implementation |
|---|---|
| Receipt number allocation | PostgreSQL sequence, NO CACHE NO CYCLE |
| Receipt PDF content | Template function with snapshotted data |
| Receipt PDF writing | `writeFile()` to deterministic path |
| Receipt hash | SHA-256 of PDF bytes |
| Receipt record | INSERT with all snapshotted values |
| Donation status update | UPDATE status to "receipted" |
| Statement PDF content | Template listing donation rows |

### What Claude Code may do (advisory only)

| Activity | Example |
|---|---|
| Decide WHICH donations to receipt | "8 donations from March are unreceipted" |
| Suggest corrections before receipting | "Jane's address is missing — update it first?" |
| Summarise results after generation | "Done. 8 receipts generated (#44–#51), 7 emailed" |
| Draft thank-you messages | Separate from the receipt itself |
| Offer follow-up actions | "Tom Nguyen has no email — send manually" |

### What Claude Code must NEVER do

- Write or modify receipt PDF content
- Suggest or influence receipt numbers
- Bypass the `--confirm` flag
- Edit receipt records after creation
- Delete receipt PDFs

## NanoClaw Integration Points

### 1. Donation recording from WhatsApp

User: "Record a $500 donation from Jane for the building appeal"
Claude Code: `qnp-crm donations add 500 --contact "Jane" --fund building-appeal --method eft` → plan → confirm → execute

### 2. Receipt generation on request

User: "Send receipts for this week's donations"
Claude Code: `qnp-crm receipts batch --from 2026-03-03 --to 2026-03-09` → shows plan with all receipts → user says YES → `--confirm --send`

### 3. Proactive reporting

User: "How are we going this month?"
Claude Code: `qnp-crm reports summary --from 2026-03-01` → formats WhatsApp reply with totals and trends

### 4. Scheduled tasks (via NanoClaw task scheduler)

- Daily: check for overdue recurring donations → alert user
- Weekly: run `reports unreceipted` → remind user about pending receipts
- Monthly: run `reports summary` → send digest

### 5. Bank reconciliation (future — nc-reconcile)

`donations.bank_transaction_id` field reserved for matching against bank feed.
`recurring_donations.match_reference` field for auto-matching recurring bank transactions.

## Environment Variables

```
DATABASE_URL=postgres://nanoclaw:nanoclaw@localhost:5432/nanoclaw
NANOCLAW_DATA_DIR=/var/lib/nanoclaw          # receipt PDF storage
NANOCLAW_PERFORMER=cli:agent                 # audit log attribution
```

Organisation-specific config stored in `receipt_config` table, not env vars.

## Non-Goals (v1)

- Payment gateway integration (Stripe, PayPal) — donations recorded after payment
- Online donation forms — future web module
- Pledge management — use notes for now
- Grant management — different workflow, separate CLI later
- Multi-currency — AUD only, schema supports it for later
- Soft credits — deferred to v2
- Per-financial-year receipt numbering — global sequence is simpler and still compliant
- Campaign CRUD — campaigns are freeform strings on donations, not managed objects
